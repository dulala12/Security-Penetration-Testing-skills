# AI 辅助代码审计工作流（金和OA场景）

## 目标

在 AI 审计中控制两件事：

- 降低误报（False Positive）
- 保留可复现证据链（可交付）

## 四阶段流程

### 1) 资产建模

- 枚举入口：`aspx` / `asmx` / `ashx` / API 路由
- 枚举敏感模块：登录、文件、报表、移动端服务、后台管理
- 建立“接口 → 方法 → 数据访问层”映射

### 2) 漏洞初筛（AI）

- 用规则先扫（拼接 SQL、上传、XML 解析、会话缺失）
- 只输出“候选点”，不直接下漏洞结论
- 每个候选点必须带：文件、类、方法、参数名

### 3) 链路确认（人工+AI）

- 验证 source 是否可控
- 验证 sink 是否可达
- 验证是否存在有效防护（参数化、鉴权、白名单）
- 结论分级：
  - `CONFIRMED`：链路完整且可稳定触发
  - `SUSPECTED`：链路部分成立，缺关键条件
  - `INFO`：仅风险设计或历史线索

### 4) 修复与复测

- 修复建议优先“最小改动”
- 复测必须覆盖：原始触发、旁路参数、相邻接口
- 输出整改优先级（P0-P3）和责任归属

## 误报抑制规则

- 仅“字符串命中”不算漏洞
- 仅“延时现象”不算 SQL 注入最终证据
- 历史漏洞 PoC 不等于当前版本可利用
- 必须标注版本、路径、方法名后再输出高危

## 推荐输出格式

每条问题统一 8 字段：

1. 位置（文件/类/方法）
2. 参数（可控输入）
3. 漏洞类型
4. 证据链（source -> sink）
5. 防护状态（缺失/可绕过/有效）
6. 影响范围
7. 修复建议
8. 复测步骤
